# Reregulation Provider Registry — v1 Build Spec

A reproducible, longitudinal, cross-program registry of every authorized provider
(entity or individual) operating under a U.S. legal-services reregulation program —
ABS, regulatory sandboxes, and allied-legal-professional / paraprofessional licenses.

**v1 goal:** ship a clean, openly licensed, *auditable* registry with longitudinal
entry/exit tracking. No harm analysis yet — just the spine. The outcomes layer (v2)
and CourtListener litigation linkage (v3) build on this. The CourtListener
entity-resolution design is included here because it constrains how you should model
names from day one.

---

## 0. Design principles (don't skip these)

This dataset will be used to argue a contested empirical question, so reproducibility
and provenance are the product, not a nicety.

1. **Every derived row carries provenance:** `source_url`, `retrieved_at`,
   `scraper_version`. No field exists without a traceable origin.
2. **Raw captures are immutable.** Before parsing, store the raw HTML/PDF/JSON exactly
   as fetched, content-hashed. All tables are *derived* from snapshots and must be
   re-derivable from them. This is what lets a skeptic audit you.
3. **Longitudinal by diffing snapshots, not by trusting a "status" field.** You infer
   entries/exits by comparing successive roster snapshots over time. Status is computed,
   not scraped.
4. **Separate raw → normalized → published layers.** Dev in a relational DB; publish
   flat files. Never hand-edit published files.
5. **Neutral framing.** Columns are named for the *proxy* they measure
   (`formal_complaint_count`, not `harm`). You are building the spine others test their
   harm definitions against.

## 1. Data model (v1)

Relational in dev (SQLite or DuckDB), exported to CSV + Parquet + a Frictionless
`datapackage.json` for release. IDs are stable, opaque strings (e.g.
`prog_az_abs`, `prov_000123`).

### 1.1 `program`
One row per reform program.

| field | type | notes |
|---|---|---|
| `program_id` | str PK | e.g. `prog_ut_sandbox` |
| `jurisdiction` | str | USPS state code; join key to FIPS ref |
| `program_name` | str | "Alternative Business Structure", "Legal Paraprofessional", etc. |
| `program_type` | enum | `abs` \| `sandbox` \| `alp_license` \| `paraprofessional_pilot` \| `community_justice_worker` \| `document_preparer` |
| `regulator` | str | e.g. "Utah Office of Legal Services Innovation" |
| `regulator_url` | str | landing page |
| `authorizing_rule` | str | citation, e.g. "ACJA §7-210" |
| `launch_date` | date | first authorization/licensing |
| `program_status` | enum | `active` \| `sunset` \| `proposed` \| `paused` |
| `sunset_date` | date null | e.g. WA LLLT; UT sandbox horizon |
| `allows_nonlawyer_ownership` | bool | Rule 5.4 relief |
| `allows_upl_waiver` | bool | distinguishes UT (ABS+UPL) from AZ (ABS-only) |
| `allows_software_provider` | bool | flags the AI-relevant programs |
| `source_url` / `retrieved_at` / `scraper_version` | — | provenance |

### 1.2 `provider`
One row per authorized entity OR individual practitioner. Slowly-changing fields
(status) live in `provider_status_event`, not here.

| field | type | notes |
|---|---|---|
| `provider_id` | str PK | |
| `program_id` | str FK | |
| `provider_type` | enum | `entity` \| `individual` |
| `legal_name` | str | as registered |
| `normalized_name` | str | computed (see §3.2) — used for matching |
| `jurisdiction` | str | USPS |
| `authorization_date` | date | first seen authorized |
| `current_status` | enum | computed: `active` \| `exited` \| `suspended` \| `revoked` \| `unknown` |
| `practice_areas_raw` | str[] | program's own terms ("family law", "landlord-tenant") |
| `practice_areas_list` | str[] null | optional mapping to JusticeBench LIST codes for cross-program comparability |
| `ownership_structure` | json null | entities only: lawyer % / nonlawyer % / capital source if published |
| `uses_technology` | bool null | from public materials |
| `uses_ai` | bool null | the column that ties back to the UPL/LLM question |
| `website` | str null | useful for the v3 match + tech/AI coding |
| `first_seen_snapshot_id` / `last_seen_snapshot_id` | str FK | bounds of observation |
| `source_url` / `retrieved_at` / `scraper_version` | — | provenance |

### 1.3 `provider_status_event`
The longitudinal heart. Generated by diffing snapshots.

| field | type | notes |
|---|---|---|
| `event_id` | str PK | |
| `provider_id` | str FK | |
| `event_date` | date | snapshot date the change was first observed |
| `event_type` | enum | `authorized` \| `status_change` \| `disappeared_from_roster` \| `disciplined` \| `reinstated` |
| `new_status` | enum | mirrors `provider.current_status` values |
| `detail` | str null | |
| `source_snapshot_id` | str FK | which capture revealed it |

Note: `disappeared_from_roster` ≠ `revoked`. A provider dropping off a roster is an
*observation*, not a legal conclusion; keep them distinct and let analysts decide.

### 1.4 `provider_alias`
Critical for entity resolution. Seed it early.

| field | type | notes |
|---|---|---|
| `provider_id` | str FK | |
| `alias_name` | str | DBA, former name, "doing business as", website brand |
| `alias_source` | str | roster / website / litigation / manual |

### 1.5 `source_snapshot`
| field | type | notes |
|---|---|---|
| `snapshot_id` | str PK | |
| `program_id` | str FK | |

### 1.6 (v3 stub) `crosswalk_courtlistener`
Defined now so the schema is forward-compatible.

| field | type | notes |
|---|---|---|
| `provider_id` | str FK | |

### Controlled vocabularies / reference data
- `jurisdiction`: USPS + FIPS crosswalk (adopt the JusticeBench states/counties/FIPS sheet).
- `practice_areas_list`: JusticeBench **LIST** taxonomy codes — gives instant interop
  with the broader A2J tooling ecosystem.
- Keep all enums in a single `enums.py` validated with `pydantic`.

### Practitioner discipline (v2 source, but capture URLs now)
Each state bar exposes a member/discipline lookup. Inventory them alongside the rosters;
they're the practitioner-side outcomes for v2.

### Program-status layer **[A]**
Keep `program.program_status`, `launch_date`, `sunset_date` current without manual checks:
- **Open States v3** — `https://v3.openstates.org/`, `X-API-KEY` header; free key.
- **LegiScan** — free tier 30k queries/mo; `getSearch` / `getMasterList`.
Query each state for reregulation-relevant bills/rules and surface status changes
(Texas going live, a sandbox narrowing, a new state entering). Open States also publishes
**open-source scrapers** — read them as your model for maintainable multi-site scraping.

## 3. Entity resolution → CourtListener (v3 design)

The hardest and most valuable layer. Goal: link each `provider` to the dockets where it
appears, so analysts can study litigation footprint and outcomes — the thing the Arizona
Republic did by hand for a handful of firms, done systematically.

### 3.1 Access pattern
- API: `https://www.courtlistener.com/api/rest/v4/`, token auth
  (`Authorization: Token <token>`). Free limits ≈ 5/min, 50/hr, 125/day — too slow for
  bulk matching.
- **Therefore match against CourtListener BULK DATA offline** (quarterly docket/party
  CSVs) and reserve the live API for verification of candidate hits. This sidesteps rate
  limits and makes the match reproducible.

### 3.2 Name normalization (`normalized_name`)
Deterministic, version it:
1. lowercase, strip accents, collapse whitespace, drop punctuation.
2. remove corporate suffixes: `llc, l.l.c., pllc, pc, p.c., inc, llp, ltd, co`.
3. drop leading `the`; standardize `&`→`and`.
4. keep an *un*stripped variant too — sometimes the suffix disambiguates.

### 3.3 Pipeline
1. **Candidate generation (blocking).** For each provider, query bulk party/docket data
   for fuzzy name hits. Block on normalized-name token overlap + jurisdiction (provider
   state ↔ court state) to cut the comparison space.
2. **Scoring.** Per candidate compute:
   - `rapidfuzz` token-set ratio on `normalized_name` (primary).
   - jurisdiction agreement (court state == provider state) → boost.
   - temporal plausibility (docket date ≥ `authorization_date`, with a grace window) → boost.
   - party-role sanity (provider as party/represented entity, not just mentioned).
   Combine into a single 0–1 `match_score` (start with a transparent weighted sum; only
   move to a trained `recordlinkage`/`dedupe` model once you have labeled pairs).
3. **Triage by threshold.**
   - high (e.g. ≥ 0.92 + jurisdiction match) → auto-accept (`match_method=fuzzy`, `verified=false`).
   - mid band → **human review queue** (this is where your legal judgment is the moat).
   - low → reject, but log for recall auditing.
4. **Persist** to `crosswalk_courtlistener` with score + method. **Manual verifications are
   immutable** — re-runs never overwrite `verified=true` rows.
5. **Measure yourself.** Hand-label a gold set (e.g. 200 pairs) to report precision/recall.
   For a contested dataset, publishing your match error rates is a credibility multiplier.

### 3.4 Known failure modes (document them)
- Common/generic names ("Legal Solutions LLC") → many false positives; lean on jurisdiction
  + date + website cross-checks.
- Brand ≠ registered name → seed `provider_alias` from each provider's website before matching.
- Pre-authorization litigation and post-name-change filings → temporal filter must be a
  documented choice, not silent.

## 4. Tech stack

- **Language:** Python 3.12.
- **Fetch:** `httpx` (static + API), `playwright` (JS directories).
- **Parse:** `selectolax` or `beautifulsoup4` (HTML); `pymupdf` / `pdfplumber` (PDF tables).
- **Validate:** `pydantic` models == the schema; fail loudly on drift.
- **Store (dev):** SQLite or DuckDB; raw snapshots on disk or S3 keyed by `content_sha256`.
- **Transform:** `polars` (or pandas).
- **Match:** `rapidfuzz`, then `recordlinkage`/`dedupe` if warranted.
- **Schedule:** GitHub Actions cron → periodic re-scrape → snapshot → diff → status events.
- **Publish:** CSV + Parquet + Frictionless `datapackage.json` + a written data dictionary
  and methodology doc.

### Repo layout
```
reregulation-registry/
  scrapers/            # one module per source; shared base class (fetch→snapshot→parse)
  models/              # pydantic schema + enums
  resolve/             # normalization + CourtListener matching
  pipeline/            # snapshot diffing → status_events; exporters
  data/
    raw/               # immutable snapshots (git-lfs or external store)
    db/                # dev database
    release/           # published CSV/Parquet/datapackage
  docs/                # data dictionary, methodology, per-source notes
  tests/               # parser fixtures from saved snapshots
```

## 5. Milestones (v1)

- **M1 — Skeleton + one vertical slice.** Schema, `pydantic` models, snapshot/provenance
  harness, base scraper class, and **Arizona ABS** end-to-end (fetch → snapshot → parse →
  validated CSV). Proves the spine.
- **M2 — Breadth.** All entity-reform rosters (AZ ABS, UT sandbox) + ALP rosters
  (AZ LP, UT LPP, CO LLP, OR, MN, WA-historical). Handle the PDF and headless cases.
- **M3 — Longitudinal.** Scheduled re-scrapes; snapshot diffing → `provider_status_event`;
  **backfill history via Wayback** to push the time series before your start date
  (essential for the WA sunset analysis).
- **M4 — Status layer + release.** Open States/LegiScan program-status automation;
  data dictionary + methodology; tag a versioned release; open-license it (CC BY 4.0 data /
  MIT code) and bring v1 to IAALS and the Stanford Rhode Center as a contribution to
  evaluations they already run.

Each milestone is independently citable — even M2 alone is a cross-program authorized-provider
census that does not currently exist in clean form.

## 6. Scraping ethics / legal footing

- These are public government records — strong footing — but still set a descriptive
  User-Agent, honor `robots.txt`/ToS, rate-limit, and cache (work from snapshots, not live
  re-hits).
- **Never scrape PACER directly** for the litigation layer — use RECAP/CourtListener.
- For bar discipline lookups, check each site's ToS; some prohibit bulk access — prefer
  any published reports/exports over hammering a search UI.
- Publish the methodology that documents every choice (temporal windows, status inference,
  match thresholds). On a contested topic, transparency *is* the defense.
